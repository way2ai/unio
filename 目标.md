**如何落地检查升级功能？**

- `/update` slash command、CLI 命令和 TUI 顶部版本卡片共用。
- 通过 GitHub Releases latest API 获取最新版本，和当前 `env!("CARGO_PKG_VERSION")` 对比。
- v1 只做“检查 + 展示 + 提示升级命令”，不自动替换当前二进制，避免 Windows/macOS/Linux 自更新差异带来的风险。
- 检查结果缓存到 state DB 或 state JSON，默认 6 小时刷新一次；用户主动执行 `/update` 时允许强制刷新。

## daemon (守护进程)

- 让系统持续活着（24小时运行的后台）

## agent（大脑）

- 直接回复（不需要调用工具或者plan或者sub agent）,表示执行完毕回复用户
- plan：复杂的事务是调用 plan 工具(规划好步骤，每个步骤交个一个 sub agent 处理)
- sub agent：处理具体的事情，sub agent 的上下文不会被引用到主 agent 对话的上下文中

**agent 工作模型：Agent Loop**
观察 → 思考 → 行动 → 反馈

**轮数增长后，如何避免模型上下文溢出？**

- 增加上下文预算配置：最大 token、压缩阈值、保留最近轮数。
- 在每轮模型请求前估算上下文占用，并按规则压缩历史。
- 压缩策略：保留 system prompt、当前用户输入、最近 N 轮完整消息；更早历史压缩成长摘要；工具输出只保留摘要、关键路径、错误和引用。
- token 占比超过 70% 开始生成/刷新摘要，超过 85% 强制压缩，超过 90% TUI 用红色提示。
- sub agent 的完整上下文不进入主 agent，只回传最终摘要、关键文件、风险和结果。

**sub agent 具体如何落地?**

- sub agent 由 root agent 或 planner 调度，用于执行明确、边界清晰的子任务。
- 子 agent 使用独立 `agent_id`，通过 `parent_agent_id` 关联 root agent；`conversation_id` 使用当前用户请求链路 ID。
- 权限策略继承父会话，可降低不可提升；例如父会话是 default，sub agent 不能升级到 auto 或 full-trust。
- sub agent 最终只返回结构化结果，不把完整上下文注入主 agent。

## [内置工具](内置tool设计.md)

1. glob 文件通配符检索，遍历项目文件树
2. grep 全局代码正则搜索、关键词查找
3. read：读取任意文件、分段读、预览目录
4. edit：差分精准编辑代码（最核心，智能补丁修改）
5. write：新建 / 覆写完整文件
6. bash：安全沙箱执行终端命令（git/test/build/install）
7. fetch：极简网络拉取公开文档源码
8. plan：任务拆解规划内部工具
9. skill-tool：每个 skill 均为一个 tool

**plan 具体如何落地?**

- `plan` 是只读内部工具，只做任务拆解和执行策略规划，不直接读写文件、不执行命令、不联网。
- root agent 判断任务复杂时调用 `plan`，输出结构化 `PlanSpec`。
- planner 拆分步骤时标注依赖关系、是否可并行、执行主体是 root agent 还是 sub agent。
- plan 结果写入 trace，并通过 daemon 事件推送给 TUI，展示当前处于 planning 阶段。
- 后续执行阶段由 root agent 根据 plan 调度工具或 sub agent。

## skill-tool

- 从 `{workspace}/.unio/skills/` 和 `~/.unio/skills/` 下读取 skills
- 根据 SKILL.md 文档，每个技能作为 tool 注入到模型入参的 tool 列表中，tool 的 name 就是技能名称，tool 的描述就是技能描述，parameter 就是当前整个上下文。
- skill-tool 的内部运行是一个单独的 skill-tool-agent，整个 skill-tool-agent 返回的最终结果就是 tool result。

**skill-tool-agent 具体如何落地？**

- skills discovery 从当前 `.skills` 目录调整为 `{workspace}/.unio/skills/`、`~/.unio/skills/`，不再保留 `.skills` 。
- 每个 skill 根据 `SKILL.md` 注入为一个模型 tool，tool name 为 skill 名称，description 为 skill 描述。
- 模型调用某个 skill tool 时，启动独立 `skill-tool-agent`。
- skill agent 先读取对应 `SKILL.md`，再按技能说明完成任务，最终结果作为 tool result 返回给主 agent。
- 调用 skill-tool 本身需要审批；skill agent 内部工具审批策略继承当前会话策略。

## 安全策略

**3大安全策略**
Sandbox（沙箱）：控制 “能碰哪些文件 / 网络”
Approval（审批）：控制 “要不要用户点确认”

1. default（默认策略）
   1. 沙箱 read-only；
   2. 免审批：read，glob，grep,plan；
   3. 禁止：write，bash,edit，fetch(修改/执行/网络都拦截)；
   4. 路径：只读当前workspace，不可读系统路径；
   5. 审批：onrequest(所有写/执行[write/edit/bash/fetch]都需要人审批，即手动同意)、要运行 skill-tool 需审批；
   6. skill-tool：skill-tool-agent 运行中内部的工具审批机制也为 default；
   7. 总结：只能看，不能改，不能跑，不能上网。
2. auto
   1. 沙箱 workspace-write；
   2. 免审批：read，glob，grep,plan，edit（增量修改），write(工作区内小于500kB非敏感文件)、bash(安全命令免审核，比如git status/diff等等)，fetch拉取白名单互联网网络；
   3. 审核：工作区以外的需要审核；大文件覆写需要审核；高危命令需要审核（rm，sudo,curl未知url等等）；要运行 skill-tool 需审批；
   4. skill-tool：skill-tool-agent 运行中内部的工具审批机制也为 auto；
   5. 总结：工作区内全自动，越界/高位必须确认。
3. full-trust
   1. 沙箱：danger-full-access；
   2. 审批：永不需要审批；
   3. skill-tool：skill-tool-agent 运行中内部的工具审批机制也为 full-trust；
   4. 总结：完全放权，风险自负。

**沙箱落地**
沙箱提供给 bash 工具使用，用于保证执行脚本时宿主机的安全性。

1. linux：Bubblewrap（基于 Linux Namespaces 命名空间和 seccomp 过滤系统调用）
2. windows：
   1. 官方原生Sandbox（基于 Hyper-V 虚拟化技术）
   2. 其他window版本：提示无Sandbox环境。无论什么审批策略都需要用户允许后才能执行。
3. macos：官方原生Sandbox（基于 Seatbelt 机制）
4. unio 如果放入容器中运行，可以添加环境变量UNIO_SWITCH_SANDBOX=0关闭沙箱，减少沙箱嵌套沙箱带来的性能损耗

## 数据持久化

- 使用 jsonl 文件保存会话历史数据
- 通过 /resume 命令可以恢复指定的会话
- 存在会话id（代表一个会话） 和 对话id（代表一次对话）

主链路：
user -> agent -> [planner <--> subagent] -> user
planner:step1 -> step2 -> ... -> stepn
subagent:skills -> tool -> approval

llm:
messages,reasoning_content,input tokens,output tokens,tool_calling

[]:表示可无
箭头表示方向

**具体如何落地？**

- SQLite 保存 session 元数据；JSONL 保存会话消息、trace、工具调用明细。
- transcript JSONL 扩展为 message-level schema，记录 user、assistant、tool、reasoning、token_usage、tool_calls。
- `/resume` 先从 `SessionStore` 找 session，再加载该 session 的 transcript；超过上下文预算时加载摘要和最近消息。
- 统一 ID 语义：`session_id` 表示会话，`conversation_id` 表示一次用户请求链路，`run_id` 表示一次 agent 执行，`agent_id` 区分 root/planner/sub/skill agent。
